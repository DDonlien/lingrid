## 开源 / 免费优先

| 软件 | 类型 | 适合场景 |
|---|---|---|
| **Lokalize** | KDE 的 CAT/本地化工具 | 我会优先推荐这个作为 Poedit 替代。支持 PO 编辑、翻译记忆、术语表、QA、项目管理；KDE 官方也列出 Windows/macOS/Linux。([KDE Applications](https://apps.kde.org/lokalize/?utm_source=chatgpt.com)) |
| **GNOME Translation Editor / Gtranslator** | GNOME PO 编辑器 | Linux/GNOME 用户很合适，专门处理 gettext PO，有查找替换、翻译记忆、消息表等。([wiki.gnome.org](https://wiki.gnome.org/Apps/Gtranslator?utm_source=chatgpt.com)) |
| **OmegaT** | 开源 CAT 工具 | 更偏职业翻译流程，翻译记忆强，跨平台；GNU gettext 文档说明它支持 PO 文件。([gnu.org](https://www.gnu.org/software/gettext/manual/html_node/OmegaT.html?utm_source=chatgpt.com)) |
| **Weblate** | 开源 Web 翻译平台 | 适合团队协作、Git 仓库同步、多人翻译。Weblate 官方支持 GNU gettext PO。([Weblate](https://docs.weblate.org/en/latest/formats/gettext.html?utm_source=chatgpt.com)) |
| **Virtaal** | 老牌开源离线翻译工具 | 支持 PO/XLIFF，但项目比较老，官网最新稳定版还是 0.7.1；不建议新项目首选。([virtaal.translatehouse.org](https://virtaal.translatehouse.org/?utm_source=chatgpt.com)) |
| **VS Code + gettext 插件** | 代码编辑器插件 | 适合开发者顺手改少量条目；主要是语法高亮，不如专用 PO 编辑器安全。([Visual Studio Marketplace](https://marketplace.visualstudio.com/items?itemName=gettext.gettext&utm_source=chatgpt.com)) |
| **GNU gettext / Translate Toolkit** | 命令行工具 | 不是 GUI 编辑器，但适合检查、合并、转换、编译 `.po/.mo`。Translate Toolkit 提供检查和管理 PO/XLIFF 的工具。([PyPI](https://pypi.org/project/translate-toolkit/?utm_source=chatgpt.com)) |

## 最大亮点：多语言矩阵视图

你这个功能非常值得做。

传统 `.po` 文件本身通常是“一个语言一个文件”，核心结构围绕 `msgid → msgstr`；GNU gettext 官方文档也把 PO 文件描述为 textual editable files，并包含 entry、workflow flags、context、plural forms 等结构。([gnu.org](https://www.gnu.org/software/gettext/manual/html_node/PO-Files.html?utm_source=chatgpt.com))

所以你的矩阵可以这样抽象：

| key | context | source | zh-CN | en | ja | flags | tags | note |
|---|---|---|---|---|---|---|---|---|
| `menu.start` 或 hash | `UI` | Start Game | 开始游戏 | Start Game | ゲーム開始 | fuzzy | todo | ... |

但注意：**PO 的唯一键不应该只是 id 或 source。**

更安全的 key 应该是：

```ts
TranslationKey = msgctxt + msgid + msgid_plural
```

因为 PO 支持 context 和 plural forms；复数形式在 gettext 里是专门结构，包含 `msgid_plural` 和多个 `msgstr[n]`，不能粗暴压平成一个字符串。([gnu.org](https://www.gnu.org/software/gettext/manual/html_node/Translating-plural-forms.html?utm_source=chatgpt.com))

## 需要特别小心的几个技术坑

### 1. “直接修改并保存打开的文件”很容易出事故

`.po` 可以做到相对安全；但 `.xlsx` 保存就复杂一些。

SheetJS 文档说明它能从 bytes / Buffer / ArrayBuffer 读取表格，也支持本地文件读写，但本地文件访问依赖运行环境；在 Electron 里问题不大，但要自己处理文件系统权限、覆盖保存、备份等。([docs.sheetjs.com](https://docs.sheetjs.com/docs/solutions/input/?utm_source=chatgpt.com))

建议：

- PO：允许直接覆盖保存
- CSV：允许直接覆盖保存
- XLSX：第一版建议“另存为”优先，直接覆盖前自动备份
- 所有保存动作生成 `.bak` 或保存历史

### 2. PO 文件不要只当普通表格

需要保留这些字段：

- `msgctxt`
- `msgid`
- `msgid_plural`
- `msgstr`
- `msgstr[n]`
- translator comments
- extracted comments
- references
- flags，例如 `fuzzy`
- previous msgid
- obsolete entries

Weblate 文档也提到 PO 里可以包含 previous source strings，例如 `#| msgid ...`，用于展示源文本变化。([Weblate](https://docs.weblate.org/en/latest/formats/gettext.html?utm_source=chatgpt.com))

所以内部数据结构一定要比表格复杂，表格只是 UI 投影。

### 3. Excel / CSV 需要定义自己的项目规范

不要尝试“兼容所有 Excel 翻译表”。第一版应该规定模板，例如：

| id | context | source | zh-CN | en | ja | note | tags |
|---|---|---|---|---|---|---|---|

否则导入逻辑会变成灾难。

可以支持导入时手动映射列：

- 哪列是 source
- 哪列是 id
- 哪列是语言
- 哪列是备注
- 哪列是 tag

### 4. Electron 安全边界要一开始做好

Electron 官方推荐通过 preload script 安全暴露特权 API 给 renderer；context isolation 的目的就是让 preload/Electron 内部逻辑和网页上下文隔离。([Electron](https://electronjs.org/docs/latest/tutorial/context-isolation?utm_source=chatgpt.com))

所以结构建议：

```txt
main process
- 文件读写
- 最近项目
- 系统菜单
- 原生 dialog
- 安全存储 API key

preload
- expose file/project/api methods

renderer
- React UI
- shadcn/ui
- table editor
- AI suggestion panel
```

不要让 renderer 直接拿 Node 文件系统权限。
