## 对话与行动记录

- 用户指出"AI 和 DeepL 不通"，先让诊断日志返回更详细信息。
- 我提议了"加 ProviderHttpError + 多个事件"和"在 catch 里拼 error message"两套方案；用户明确说"没这么复杂"，只让把请求/输出/返回都打到现有诊断日志。
- 决定保留项目里现有 `appendDiagnostic(event, details)` 模式，新增 4 个事件族：`ai.request.start` / `ai.request.success` / `ai.request.error` 与 `deepl.request.start` / `deepl.request.success` / `deepl.request.error`，加上批处理路径的 `ai.batch.item.error`。
- 改 `ts/renderer/providers/openai-compatible.ts`：
  - 引入 `ProviderHttpError` 类（带 `status` / `contentType` / `bodyPreview` 字段）
  - 新增 `safeReadBody` helper：截断到 600 字符 + `…(truncated)` 标记
  - `requestOpenAiCompatibleTranslation` 在 `!response.ok` 时读 body 抛 ProviderHttpError
- 改 `ts/renderer/providers/deepl.ts`：同上结构，对 `DeepLResult` 加可选 `rawBody` 字段（暂未用，保留扩展位）。
- 改 `ts/renderer/App.tsx`：
  - import 两个 provider 的 `ProviderHttpError` 别名
  - 新增 `describeAiError(error)` helper：识别 `ProviderHttpError` 返回 `{ status, contentType, bodyPreview }`，其他 Error 返回 `{ name, message }`
  - 新增 `aiErrorNotice(provider, error)` helper：拼 `AI request failed (HTTP 403: Wrong endpoint, please check your DeepL account region.)` 这种句子给 status bar
  - 新增 `extractServerMessage(bodyPreview)` helper：从 DeepL / OpenAI 错误体 JSON 抽 `message` 字段；非 JSON 且长度 < 200 则原样返回
  - `requestAiTranslation` 内三处 try/catch：成功路径打 `*.request.success`、失败路径打 `*.request.error` 并把 `describeAiError(error)` 展开；batch 路径用 `ai.batch.item.error` 逐条留痕
  - `generateSuggestion` 的 catch 改为 `setNotice(aiErrorNotice(ai.provider, error))`，把具体状态码和 server message 显示给用户
- 加 7 个 vitest 单测（`ts/tests/providers.test.ts`）：
  - OpenAI 2xx 返回 strip 后的 text
  - OpenAI 4xx 抛 `ProviderHttpError`，`bodyPreview` 含 server message
  - OpenAI 500 大 body（2000 字符）被截断到 < 700 字符且包含 `truncated`
  - DeepL 2xx 返回 translation + detectedSource
  - DeepL 403 forbidden（典型 region 配错场景）抛带 `403 Forbidden` 的 Error
  - DeepL 400 `target_lang not supported` 抛带 `400` 的 Error

## 备注

- **本轮未做** "AI 不通的根因"——根因排查属于下一轮（等用户拿真 key 复现一次，让诊断日志能告诉我们具体哪步挂、什么状态码、什么 server message）。本轮只把诊断能力建到位。
- **body 截断阈值 600 字符**是个保守值，DeepL / OpenAI 错误体通常 < 200 字符；遇过 Cloudflare / 反代大 body 的可能不够，可在以后调。
- **`ProviderHttpError` 在两个 provider 文件里各自定义**——`openai-compatible.ts` 和 `deepl.ts` 互不引用。App.tsx 用 `import as` 重命名两个类后用 `??` fallback。**简单方案**，但稍微 hacky；以后如果再添 provider，**建议**提到 `providers/errors.ts` 单文件。当前不动，避免引入跨文件改动。
- **apiKey 不入日志**——遵循 `AI-A-009` 与项目历史约定；新事件细节字段只记 endpoint / model / target_lang / status / bodyPreview，**不**记 `apiKey` / 完整 prompt / 译文正文。
- **本轮没有触发 git 操作**——所有改动都在 working tree，未 commit 也未 push。
- vite dev server 与 playwright 没有启动（本轮不需要 e2e 验证）。
